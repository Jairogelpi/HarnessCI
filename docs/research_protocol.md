# HarnessCI Research Protocol

## Main research question

Can a hybrid audit based on specification, diff, tests, and harness traces detect risks in agent-generated Pull Requests better than tests alone?

## Hypotheses

- **H1:** Agent-generated PRs can pass tests while partially violating the specification.
- **H2:** Diff minimality and spec compliance predict PR acceptability better than test pass/fail alone.
- **H3:** Harness traces such as tool calls, retries, edit attempts, and failed test runs predict integration risk.
- **H4:** Different agents or harnesses produce different risk profiles on the same task.
- **H5 optional:** A combined risk score prioritizes human review better than simple metrics such as files changed or tests passed.

## Dataset: AgenticPR-Bench-mini

Recommended TFM dataset size:

```text
15-20 tasks x 2-3 agents/harnesses x 3 repetitions = 90-180 PRs
```

A smaller defensible version may use 60-100 PRs if time is constrained.

## Task types

- simple bugfix;
- bugfix with edge case;
- small feature;
- controlled refactor;
- test generation;
- security-sensitive change;
- API behavior change;
- database migration;
- frontend validation;
- documentation update.

## Base repositories

Use three toy-but-realistic repositories:

1. `fastapi-auth-demo`
2. `django-billing-demo`
3. `react-dashboard-demo`

Each repository should include issues, task specs, tests, ground truth patch, expected touched files, out-of-scope areas, and risk labels.

## Gold labels

Each PR receives one primary label:

- `ACCEPTABLE`
- `NEEDS_REVIEW`
- `UNACCEPTABLE`

And boolean attributes:

- `spec_violation`
- `unrelated_changes`
- `missing_tests`
- `security_sensitive`
- `overengineering`
- `architecture_drift`

Initial labels may be created by the researcher. If possible, ask 1-2 additional reviewers to label a sample and calculate agreement.

## Baselines

Compare HarnessCI against:

1. tests pass/fail only;
2. number of files changed or churn threshold;
3. simple static risk rules;
4. HarnessCI combined risk score.

## Metrics

Technical evaluation: precision, recall, F1, false positive rate, false negative rate, and ROC-AUC if sample size allows.

Engineering and review metrics: spec compliance, diff minimality, test adequacy, security risk, architecture drift, and review priority.

Harness metrics: tokens, tool calls, test runs, edit attempts, retries, latency, and cost estimate.

Final product metrics: risk detection rate, unsafe-pass detection, review prioritization quality, and cost-adjusted PR quality.

## Evaluation claim

The thesis should not claim perfect automatic code review. It should show whether a transparent hybrid audit adds measurable value over tests-only validation for PRs generated by coding agents.
