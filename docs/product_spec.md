# HarnessCI Product Specification

## Product identity

**Name:** HarnessCI  
**Academic name:** Forensic auditing of LLM-agent-generated Pull Requests using specifications, harness traces, and risk analysis  
**Tagline:** Stop merging AI-generated PRs blindly.

HarnessCI is an open-source CI tool for auditing Pull Requests generated by coding agents before a maintainer spends time reviewing them or before they enter the repository.

## Problem

AI coding agents can now create complete Pull Requests, but those PRs can still:

- pass tests while missing the original intent;
- fix a bug by touching too many unrelated areas;
- introduce technical debt, hidden behavior changes, unsafe logic, or architecture drift;
- invent internal APIs or modify security-sensitive flows;
- reduce coverage or avoid meaningful tests;
- consume excessive tool calls, retries, tokens, or failed test runs for a simple task;
- waste more reviewer time than manual implementation would have taken.

Traditional code review tools ask, "Does this code have problems?" HarnessCI asks a more specific question: "Did this agent-generated PR do what the issue asked, with the smallest reasonable diff, without hidden risk, and with a reasonable generation process?"

## Product outcome

HarnessCI will install as a GitHub Action and comment on each Pull Request with an audit decision:

- `PASS`
- `REVIEW_REQUIRED`
- `BLOCK`
- `INSUFFICIENT_INFORMATION`

A typical report includes spec compliance, diff minimality, test adequacy, security risk, architecture drift, harness efficiency, overall agentic risk, findings, and recommendation.

## Evidence sources

HarnessCI combines five evidence sources:

1. **Original specification:** issue, PR body, task/spec files, EARS requirements, OpenAPI, README, manual acceptance criteria.
2. **Actual diff:** changed files, churn, functions/classes touched, critical areas, dependencies, migrations, public APIs.
3. **Technical execution:** tests, coverage, lint, type checking, security/static analysis.
4. **Agent/harness telemetry:** planning, reflection, memory mode, tool calls, retries, edit attempts, test runs, failed runs, token/cost estimates.
5. **Hybrid judge signal:** deterministic rules and heuristics first; optional LLM judge as one bounded input, never the sole decision maker.

## Target users

### Open-source maintainers

Maintainers receiving AI-generated PRs who need to filter noisy or risky contributions and decide whether a PR deserves human review.

### Companies with internal agents

Teams using Claude Code, Codex, Copilot Coding Agent, OpenHands, Devin, Cursor, Kilo, or similar systems who need to compare agents, harnesses, cost, reliability, and risk.

### Researchers

Researchers studying real quality of agentic PRs beyond benchmark pass/fail results.

## MVP scope

Required MVP:

- GitHub Action
- Local CLI
- Specification parser and normalizer
- Diff parser and minimality analyzer
- Configurable tests/lint/coverage/security command execution
- Rules-based risk scoring
- Markdown PR comment/report
- Experimental AgenticPR-Bench-mini dataset
- Gold labels and empirical evaluation
- Dashboard with results

Out of scope for the MVP:

- Marketplace-grade SaaS
- Enterprise authentication
- Deep support for every language
- Full integration with every agent
- Proprietary trained model
- Perfect architecture analysis
- Massive public dataset
